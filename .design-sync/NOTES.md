# Design sync notes — mixjam-electron

## Repo shape

This repo is an Electron renderer app (`src/renderer/src/components`), not a
publishable package — no `dist/`, no `module`/`exports`/real `main` for the
component set (`package.json` `main` points at the Electron main-process
build, `out/main/index.js`). The sync runs in **synth-entry mode** always;
there is no "real build" to point `--entry` at instead.

`cfg.buildCmd` in config.json passes `--entry ./out/main/does-not-exist.js` —
a **deliberately nonexistent path under the repo root**. This is required: it
makes `resolveDistEntry`'s `soft: true` branch fail and fall through to
synth-from-src, while still resolving `PKG_DIR` to the repo root (via the
walk-up-to-package.json logic) so `cfg.cssEntry`/`cfg.tsconfig`/`cfg.extraFonts`
(all resolved relative to `PKG_DIR`) find the real files. Do not "fix" this by
pointing `--entry` at a real file — any real file triggers the hard-override
path and skips synth discovery entirely (skips `deriveComponentsFromSrc`).

## Known render warns (checked against on every re-sync)

- `[RENDER_THIN] ScanOverlay: rendered height is 0px` — `.scan-overlay` is
  `position: fixed; inset: 0`. The local render-check/capture tooling measures
  document/element height, which a fixed-position overlay reports as 0 even
  though it paints centered content correctly (confirmed by reading the actual
  screenshot — title/phase/progress bar all present and themed, just cropped
  at the top in the review capture image). This does not reflect how the real
  claude.ai/design product renders the card (full declared viewport, `900x700`
  via `cardMode: single`). Benign — do not chase.
- `[RENDER_THIN] AppShell: variants render identically` — false positive. The
  8 per-theme AppShell stories share identical DOM structure (same Header +
  TrackerView + Footer, same lanes/layout); only the CSS custom property values
  differ per theme. That heuristic doesn't diff computed colors, so it reads the
  structural sameness as "identical". The captured screenshots show 8 visibly
  distinct themes. Benign — do not chase.
- `[FONT_DANGLING]` on Josefin Sans / Ubuntu / JetBrains Mono — the repo's
  `index.css` uses root-relative `@font-face` URLs (`/fonts/Foo.ttf`), which
  the *local* validator's naive `resolve(dirname(cssFile), url)` mistakes for
  an absolute filesystem path (never resolves) even though the font files are
  genuinely present under `ds-bundle/fonts/` (confirmed by `ls`). Non-blocking
  scrape-only false positive; do not chase.
- `[TOKENS_MISSING]` doesn't fire anymore — see "Theme tokens are real,
  static CSS now" below. If it reappears, `cfg.cssEntry` probably got pointed
  back at bare `index.css` instead of the generated
  `.design-sync/design-sync-styles.css`.

## Theme tokens are real, static CSS now (not just runtime-injected)

**This was a real gap the first pass missed, caught in human review.** MixJam
ships 8 themes as JSON files (`public/themes/*.json` — several were
new/untracked in git when this sync started). The running app applies them
via `theme/themes.ts`'s `applyTheme()`, which calls
`root.style.setProperty(...)` at runtime — so there is no static CSS anywhere
that defines, say, Neon Rave's colors. Syncing only `index.css` (the original
`cfg.cssEntry`) shipped a design system where only the default Emerald theme's
*bootstrap* values existed anywhere, and only implicitly (as the values the
provider happens to set on mount) — the other 7 themes had no representation
in the uploaded project at all, which the user (correctly) flagged as a risk
of losing that data if the old project's mockup HTML files (which had at
least baked-in per-theme snapshots) were deleted.

Fix: `.design-sync/generate-theme-tokens.mjs` reads all 8
`public/themes/*.json` files and emits
`[data-theme-key="<key>"] { --token: value; ... }` blocks — real, static,
`var(--*)`-discoverable CSS — for every theme, matching exactly how
`theme/themes.ts` names and structures its tokens. Since `cfg.cssEntry` is
copied **verbatim** by the converter (it does not resolve `@import`s — an
early attempt at `@import`-ing the generated tokens from a wrapper CSS file
shipped a broken unresolved `@import` in `_ds_bundle.css`, tripping both
`[CSS_PLACEHOLDER]` and `[CSS_IMPORT_MISSING]`), the generator instead **concatenates**
`index.css` + the 8 generated blocks into one flat file:
`.design-sync/design-sync-styles.css` (gitignored — see below; regenerated by
`cfg.buildCmd`, which now runs the generator before `package-build.mjs`).
`cfg.cssEntry` points at that concatenated file, not bare `index.css`.

`public/themes/*.json` is the single source of truth this script reads from —
it never invents or hand-transcribes values. Re-run
`node .design-sync/generate-theme-tokens.mjs` any time a theme JSON changes;
`cfg.buildCmd` already does this before every build, so a plain re-sync
picks up theme edits automatically.

`design-sync-styles.css` itself is **generated, not committed** — add it to
`.gitignore` alongside `.ds-sync/`/`ds-bundle/` (only
`generate-theme-tokens.mjs`, the script, is durable/committed).

Two story-authoring gotchas from proving this actually works (added
`NeonRaveTheme`/`RustIndustrialTheme` stories to Header/TrackerView to verify
non-default themes render, not just exist in the CSS):

- Overriding the theme for a single story means calling `selectTheme()` from
  inside that story's own wrapper — but it must be a plain `useEffect`, not
  `useLayoutEffect`. `cfg.provider` (`ThemeBootstrap`) wraps every card as the
  outermost component and also calls `bootstrapTheme()`/`selectTheme()` in a
  layout effect; React fires layout effects child-first on mount, so a
  `useLayoutEffect` in the story-local wrapper (a descendant of the provider)
  still runs *before* the parent provider's, which then stomps the override
  back to the default theme. `useEffect` guarantees the story's call runs
  after all layout effects (including the provider's) have completed.

## Provider wiring (load-bearing — do not remove)

MixJam theming is NOT a React context/provider component — it's a plain
function (`bootstrapTheme()`/`selectTheme()` in `theme/themes.ts`) that writes
CSS custom properties onto `document.documentElement` and sets
`data-theme-ready="true"`. Critically, `index.css` has
`body { visibility: hidden }` until `html[data-theme-ready='true']` — so
**every preview is invisible by default** unless this bootstrap runs.

`.design-sync/theme-bootstrap.tsx` is a small wrapper component
(`ThemeBootstrap`, **named export — must stay named**, not default: the
extraEntries merge step in `package-build.mjs` also does a blind `export *`
that drops default exports, same root cause as the source-kit.mjs fork below)
that calls `bootstrapTheme()`/`selectTheme()` in a `useLayoutEffect`. It's
wired in via `cfg.extraEntries` + `cfg.provider.component: "ThemeBootstrap"`.
If a future re-sync's `[PROVIDER_UNEXPORTED]` fires, check this file still has
a **named** (not default) export.

## Forked lib: source-kit.mjs

`.design-sync/overrides/source-kit.mjs` forks the package adapter's
synth-entry writer. All MixJam components use `export default function X()`.
The stock synth-entry writer does `export * from "<file>"` for each source
file — standard ESM semantics mean `export *` **never** re-exports a default
export, so `window.MixJam` bundled empty (`[BUNDLE_EXPORT]`: 9/9 not on the
global) even though component *discovery* (a separate ts-morph declaration
scan) correctly found all 9 names. The fork writes named re-exports instead
(`export { default as FolderCard } from "..."`), reusing the same declaration
scan discovery already does. Declared in `cfg.libOverrides`. Needs
`.design-sync/node_modules` (junction to `.ds-sync/node_modules`) so the fork's
bare `ts-morph` import resolves — recreate on a fresh clone:
`ln -sfn ../.ds-sync/node_modules .design-sync/node_modules` (or on Windows,
a directory junction: `New-Item -ItemType Junction`).

## Grid overflow overrides

`Header`, `LaneClipCanvas`, `TrackerView`, `ManagePanel` → `cardMode: column`
(all render wider than a grid cell — chrome bars and full-page/full-panel
compositions). `ScanOverlay` → `cardMode: single` (fixed-position modal,
escapes any grid entirely; `primaryStory: "Scanning"`).

## Preview compositions worth knowing about

- `ManagePanel` previews wrap the component in a `position: relative`,
  concretely-sized host div. `.manage-panel` is
  `position: absolute; inset: 0` against a positioned ancestor with real
  height in the real app (the tracker container) — without that ancestor,
  `bottom: 0` has no reference and `.manage-content { flex: 1; min-height: 0 }`
  collapses to zero height, so the tag/library/category lists silently didn't
  render at all until this wrapper was added.
- `TrackerView` preview props were ported directly from
  `src/renderer/src/components/TrackerView.test.tsx`'s `renderTracker()`
  default prop set (the closest thing to a canonical "here's how this
  component is actually used" fixture, since there's no docs/stories tree).
- `TrackerView` also has one full-window story **per theme** (`ThemeEmerald` …
  `ThemeClubPA`, 8 stories) on top of Populated/Scanning/EmptyLibrary, kept for
  judging the tracker in isolation — see the AppShell section below for why
  per-theme full-window stories exist at all and how the two overlap.

## AppShell — the full-window preview component (design-sync only)

`.design-sync/app-shell.tsx` (exported `AppShell`) is a **preview-only harness**,
not part of the shipped app — it composes `Header` + body + `Footer` exactly as
`App.tsx` does, owning the active theme in state so the header's theme selector
works live. Added at the user's request: reviewing the whole window with a
working theme dropdown (and per-theme stories) is what the old design project's
per-theme mockups gave, which per-component previews can't.

How it's wired (three coordinated pieces — all must stay in sync):

1. It lives OUTSIDE `srcDir`, so the fork (`source-kit.mjs`) discovers it via a
   hardcoded `EXTRA_COMPONENT_FILES = ['.design-sync/app-shell.tsx']` list —
   NOT a cfg key (the config validator in `common.mjs` runs before the fork
   loads and would reject an unknown key). It is therefore NOT in `extraEntries`
   (that would double-export it → `[EXPORT_COLLISION]`).
2. Its group is `Pages` via `cfg.docsMap.AppShell` → `.design-sync/docs/AppShell.md`
   (frontmatter `category: Pages`). For that category to win, the fork also adds
   `design-sync`/`.design-sync`/`..` to `GENERIC_DIR` so the path-derived group
   doesn't beat the doc category.
3. It applies its `initialTheme` in a **`useEffect`** (not `useLayoutEffect`) —
   same provider effect-ordering reason as the `Themed` wrappers (see the theme
   section above): the `ThemeBootstrap` provider's layout effect runs after a
   child layout effect and would stomp the theme.

Do NOT add `componentSrcMap: {AppShell: ...}` — that makes `names` non-empty,
which skips the fork's `deriveComponentsFromSrc` fallback and drops all 9 real
components. `EXTRA_COMPONENT_FILES` (merged into `srcFiles`) is the right lever.

Judging a whole-page composition in every theme is a different thing from
judging components in one theme (what reads well in Emerald may not in Club
PA's black/white or Neon Rave's saturation) — this is why AppShell's 8
per-theme stories exist, replacing the old design project's per-theme static
mockup HTML files with the real component driven by real theme data. If a
theme is added to `public/themes/`, add a matching `Theme<Name>` story here
(it shows up as a token block automatically via the generator). The 8
per-theme TrackerView stories (`ThemeEmerald`…) are redundant with these for
whole-page judgment, but kept for judging the tracker in isolation.

## guidelinesGlob is deliberately empty

`cfg.guidelinesGlob: []` — the repo's `docs/` folder is engineering docs
(virtualization strategy, audio engine, SQLite schema, IPC contracts), NOT
design usage guidance. The default glob (`docs/*.md`) would copy them into
`guidelines/` and mislead the design agent. Design guidance lives in
`.design-sync/conventions.md` (the `readmeHeader`) instead. If genuine design
guides are ever added to the repo, point `guidelinesGlob` at them.

## Theme font/texture pass (2026-07-01)

User-requested per-theme identity pass, done directly in `public/themes/*.json`
(single source of truth — see "Theme tokens are real, static CSS now" above)
plus new theme-scoped CSS in `index.css`:

- **Rust Industrial** (`rust.json`): `fonts.chrome` and `fonts.label` both
  changed from Josefin Sans / Courier New to **Special Elite** (a period
  typewriter face) — Courier New/Josefin read as anachronistic against the
  rust-industrial look. `fonts.mono` stays JetBrains Mono. New
  `SpecialElite-Regular.ttf` added under `src/renderer/public/fonts/` (Google
  Fonts, OFL/Apache-licensed, fetched from the `google/fonts` GitHub repo —
  same sourcing pattern as the existing Josefin/Ubuntu/JetBrains files), with
  a matching `@font-face` in `index.css` and an entry in `cfg.extraFonts`.
- **Neon Rave** (`rave.json`) and **Screen Maximal**/**Club PA**
  (`screen.json`/`pa.json`) already used JetBrains Mono across all three font
  tokens before this pass — no change needed, just confirmed.
- **Flat Studio** (`studio.json`): `fonts.mono` changed from JetBrains Mono to
  **Ubuntu**, so all three tokens are Ubuntu — the user wants Flat Studio on
  a single family.
- **Rust Industrial texture**: `[data-theme-key='rust'] .app::before/::after`
  in `index.css` layer a tiled SVG fractal-noise grain (`mix-blend-mode:
  overlay`) plus three angled `repeating-linear-gradient` passes standing in
  for random scratches, over the whole window.
- **Screen Maximal texture**: `[data-theme-key='screen'] .app::before/::after`
  layer CRT scanlines (`repeating-linear-gradient` + the same fractal-noise
  technique for pixelated grain, both blended `overlay`, with a `steps()`
  flicker animation) and a subtle RGB-fringe VHS drift (`mix-blend-mode:
  screen`, animated horizontal jitter).
- Both overlays live on `.app::before`/`.app::after` (`.app` gained
  `position: relative` to anchor them) rather than per-surface, so every
  full-window composition (real app, `AppShell` preview) gets them for free
  without touching individual component CSS. They deliberately do **not**
  reach `.scan-overlay` (a `position: fixed` modal that already escapes
  `.app` — see the `[RENDER_THIN]` note above).
- **Gotcha hit while doing this:** the SVG data-URI noise filter's
  `xmlns='http://www.w3.org/2000/svg'` tripped
  `spec-002-theming-skin-system.test.tsx`'s `expect(css).not.toMatch(/https?:\/\//i)`
  guard (meant to catch real external font/CDN URLs, not incidental `http://`
  substrings). Fixed by percent-encoding the `://` in the xmlns
  (`http%3A%2F%2Fwww.w3.org%2F2000%2Fsvg`) — decodes identically at parse
  time, doesn't match the regex. If a future texture needs another data-URI
  SVG, apply the same encoding up front.

## WaveformPreview added (2026-07-02 re-sync)

Latest commit (`668cb85`) added `src/renderer/src/components/WaveformPreview.tsx`
(footer waveform canvas for the selected sample) — picked up automatically as
component #11 via the existing `deriveComponentsFromSrc` discovery, no config
changes needed.

- Needed an authored preview (`.design-sync/previews/WaveformPreview.tsx`):
  the component draws to a `<canvas>` via `getSampleBuffer(filepath) =>
  Promise<AudioBuffer | null>`, and jsdom has no real Web Audio API, so the
  auto-generated preview produced a blank canvas (`[RENDER_BLANK]`, PNG <5KB).
  Fixed by duck-typing a minimal `AudioBuffer`-shaped fixture (`numberOfChannels`,
  `length`, `getChannelData()` returning a synthetic sine-envelope
  `Float32Array`) — only the three members `computePeaks`/the draw loop
  actually read. Two stories: `Selected` (real waveform, bars visible in
  `--highlight` teal) and `Empty` (`filepath: null`, correctly renders nothing).
- Grade file schema note (easy to get wrong): `.design-sync/.cache/review/<Name>.grade.json`
  is `{"cells": {"<StoryName>": {"verdict": "good"|..., "note": "..."}}}` —
  NOT a flat map of `{"<StoryName>": {"grade": ...}}`. Check an existing
  component's `.grade.json` (e.g. `Footer.grade.json`) for the exact shape
  before hand-writing one.
- The other 10 components verified `unchanged` via the remote `_ds_sync.json`
  anchor (fetched with `DesignSync get_file` since local `.sync-diff.json`
  history didn't carry it) despite a commit landing after the last sync —
  the commit's font/icon/CSS changes didn't touch those components' source
  files, only shared CSS, so `styleChanged: true` still forced bundle/styling/
  fonts re-upload even with 0 component re-verifications needed.
- Re-run gotcha: calling `resync.mjs` directly (not via `cfg.buildCmd`) means
  you must pass `--entry ./out/main/does-not-exist.js` yourself — omitting
  `--entry` makes it default toward resolving `node_modules/<pkg>/package.json`
  as a real dist entry, which doesn't exist and throws `ENOENT` (this is the
  same synth-entry trick documented in "Repo shape" above, just easy to forget
  when invoking the driver ad hoc instead of via the configured `buildCmd`).

## Re-sync risks

- `docsDir`/`docsMap` unset — there's no docs tree in this repo, so every
  `.prompt.md` is synthesized from `.d.ts` + JSDoc + the authored preview.
  If per-component `.md` docs are ever added under the components dir, wire
  `cfg.docsDir` and re-run.
- The `node_modules/mixjam-electron` self-reference junction created during
  early debugging was removed (superseded by the nonexistent-`--entry` trick)
  — do not recreate it; it's not needed and its presence caused
  `pkgJson.main` (`out/main/index.js`) to wrongly resolve as a dist entry.
- No `.gitignore`/build step guarantees `out/main/index.js` doesn't
  reappear at a path the build could mistake for something real — this
  hasn't happened, but if a future `[NO_DIST]`-adjacent bug appears, re-check
  that `--entry ./out/main/does-not-exist.js` still doesn't exist.
- Rust Industrial/Screen Maximal's texture overlays (see "Theme font/texture
  pass" above) live in plain `index.css` under `[data-theme-key='...']`
  selectors, copied verbatim like everything else in `cfg.cssEntry` — no
  special wiring needed on re-sync. But they're keyed off `data-theme-key`
  the same way the generated token blocks are, so if a theme's `key` in its
  `public/themes/*.json` ever changes, update the matching selector in
  `index.css` too (the generator doesn't touch these — they're hand-authored,
  not derived from the JSON).
- Theme tokens are entirely runtime-injected (see Provider wiring above) —
  if MixJam ever adds a *second* runtime-only visual system (e.g. a
  print/export stylesheet), it will need the same `extraEntries`-wrapper
  treatment, not `cfg.tokensPkg` (there is no static tokens package).
